Part 1

Why multi-chain dApps?

Context, and the problem everyone keeps hitting

Three minutes of context. What's broken about building on-chain apps today.

The problem with "pick one chain"

  • Your users & liquidity are spread across many chains — pick one and you cut off the rest.
  • The usual fix is bridging assets over… and bridges are the #1 hack target in crypto — it's hard to do right.
  • No single chain has it all — some are slow, some expensive, some lack the assets, or the features your app needs.
  • Every chain has its own VM, wallets, tooling → supporting another chain means a rewrite.
Set up the pain. Liquidity is fragmented. The dominant cross-chain UX is asking a user to bridge, and bridges are hard to get right — they hold custody and have been drained for billions. And no single chain has everything: some are slow, some expensive, some don't have the assets or features you need. So devs either commit to one chain or rewrite for each — both bad.

What people actually want

Reach users
on every chain + Use each chain's
strengths + No bridging
No custody risk
  • EVM — deep markets & existing tokens  ·  Midnight — privacy / ZK
  • Celestia — cheap data availability  ·  Cardano — native assets

A multi-chain dApp = one app that reads from many chains and folds them into one consistent state.

Flip to what we want. Use each chain for what it's best at — EVM for markets and tokens, Midnight for privacy, Celestia for cheap DA, Cardano for native assets — without forcing users to move funds. That's the definition I'll use: one app, reads many chains, one consistent state. That's exactly what EffectStream lets you build.
Part 2

How EffectStream manages it

The short tech version

Four minutes. The core mental model and the three pieces that make it work.

The core model

Many chains
EVM · Midnight · … → Sync layer
normalize blocks → State machine
your TS logic → Indexed state
SQL / API

Read from many chains → fold into one deterministic state machine → query the result.

It's a sovereign rollup: inputs are ordered on-chain (data availability), your logic runs in your node, the resulting state is queryable.

One sentence to remember: read many chains, fold into one deterministic state machine, query the result. The "sovereign rollup" framing: the chains give you ordering and data availability — they store the inputs — but YOUR code decides what those inputs mean. Same inputs, same code, same state on every node. Deterministic.

Three pieces

Sync

Per-chain fetchers (viem for EVM, UTXORpc for Cardano, Midnight SDK, Bitcoin, Celestia…) poll finalized blocks and normalize them into one common format.

State machine

You write a typed grammar of commands + one handler each, in TypeScript. The runtime routes each input and commits SQL atomically per block.

Batcher

Users sign inputs and POST them over HTTP. The batcher packs many into one on-chain tx. Storage-first — survives crashes.

Three pieces. Sync: a fetcher per chain that turns very different blocks into one common shape. State machine: this is what you actually write — a grammar of allowed commands, a generator function per command, and every block your yielded SQL queries commit in a single transaction. Batcher: solves throughput — instead of every chess move being its own gas-paying tx, users sign and POST, the batcher bundles them into one transaction, and it persists everything to disk first so nothing is lost.

The payoff: non-custodial

  • All real interactions land on the chains, not inside your node.
  • Users keep full custody of their L1 assets the whole time.
  • “Even if your app gets hacked, users' L1 assets are not at risk.”

Plus: complex logic runs in TypeScript (not Solidity), so game state, physics, and rich rules are finally practical.

The big payoff and the differentiator versus most L2s: it's non-custodial. The framework never holds user funds — assets stay on L1. So the blast radius of a bug is your app state, not people's money. That's huge for brands and for trust. And because your logic is TypeScript, not the EVM, you can do things that are impractical on-chain — real physics, real game rules.

New, accurate picture — read it top to bottom. The chains at the top only do two jobs: order the inputs and store them (data availability). Primitives are typed watchers that pick out the events you care about. Sync normalizes all those different chains into one deterministic stream. Your TypeScript state machine folds that stream into application state — deterministic, so every node agrees. The result is queryable via a database and API. And the batcher (the pink loop) is how user inputs get on-chain efficiently: sign, bundle, one transaction, which then flows back in at the top.

How it works

Do recent indexer issues affect our indexer?

  • EffectStream's sync reads from a chain's indexer/RPC — it's decoupled from it.
  • If an upstream indexer (e.g. Midnight's) is degraded, sync slows or pauses, then resumes from the last confirmed point — state stays deterministic, nothing corrupts.
  • Live status of any current indexer issue → confirm with the team (ops question, not architectural).
Honest framing: this is partly an ops question I can't answer from the code. Architecturally, our sync service is a consumer of the chain's indexer/RPC; it tracks its own cursor and is deterministic, so an upstream hiccup delays ingestion but doesn't corrupt state — it catches up when the source recovers. For the specifics of the "recent issues," defer to the team's current status.
How it works

Sync times & slow finality (rollups settling for days)?

  • Each chain has a confirmationDepth — sync waits N blocks before treating a block as final (EVM ~12, BTC ~6, instant-finality 0).
  • A per-chain delay ≈ depth × blockTime + buffer absorbs latency & reorg risk; slow/deep-settlement chains get a higher value.
  • The deterministic NTP "Main" clock ticks independently, so the app keeps a steady tick-rate regardless of any one chain's settlement speed.
confirmationDepth (config/src/schema/sync-protocols/common.ts) sets reorg safety per chain; ordering is provisional as blocks arrive and finalized at depth. As a based rollup, EffectStream inherits its DA layer's finality. The NTP main protocol decouples app tick-rate from variable chain block/finality times.
How it works

Batcher — per chain, then sync? Not cross-network at once?

  • Correct. The batcher batches per chain / per adapter — independent queues, one bundled tx per chain — and the sync layer picks each up.
  • It is not doing a single atomic operation across multiple networks simultaneously.
  • Batch triggers: time / size / value / hybrid / custom. Confirmation: no-wait · wait-receipt · wait-effectstream-processed.
Per packages/batcher/README: each adapter target (e.g. "ethereum", "midnight") has its own queue and polling loop, so a slow chain doesn't block others — but they're independent per-chain submissions, not one cross-network atomic batch. Storage is the source of truth, so it survives restarts.
How it works

Is this mostly a metadata standard for publishing across networks?

  • No — it's a full execution + state model: read many chains → deterministic state machine → queryable state.
  • The metadata/NFT-projection standards (PRCs) are an optional feature, not the core.
Common misconception that conflates EffectStream with the PRC standards it publishes. Core = data projections + deterministic computation + optional settlement/batching. PRC-3/5 (inverse projections, serving live metadata to L1 marketplaces) are an optional layer on top, not the foundation.

Common issues devs hit with cross-chain apps?

  • Finality differences & reorgs — handled by per-chain confirmationDepth.
  • Different block times / ordering — handled by the NTP "Main" clock.
  • RPC reliability & rate limits; per-ecosystem wallet & address formats — abstracted by typed primitives + the wallets package.
The framework absorbs most of the classic cross-chain pain: reorg/finality via confirmationDepth, ordering via the deterministic clock, and wallet/format heterogeneity via @effectstream/wallets and typed primitives. RPC reliability/rate-limits are still an operational concern devs should plan for.
The template

How is the private data tied to the metadata on Ethereum?

  • Linked by a composite key: (contract_address, token_id).
  • EVM Transfer event → row in evm_midnight (ownership).
  • Midnight ledger update → property on the same token_id; a SQL join yields one record = public token + private property.
From templates/evm-midnight-v2 state-machine.ts + migrations: the transfer-assets STF writes ownership keyed by (contract_address, token_id); the midnightContractState STF writes properties keyed by the same pair. They're never bridged — the state machine just joins two chains' events on the token id.
Capabilities

Swaps across >2 chains at once (BTC ↔ ETH ↔ NIGHT)?

  • Coordination yes; true 3-way atomic, no.
  • EffectStream sequences the legs: detect on chain A → act on B → act on C. Each leg is atomic in the state machine; the overall flow is sequential — no bridge.
Honest answer. It's a multichain co-processor: it reads many chains and coordinates settlement, but it doesn't impose protocol-level atomicity across 3+ chains (that's bounded by the underlying chains' finality). The multi-chain-token-transfer template shows the sequential pattern.
Capabilities

What determines transaction order? How is time managed across networks?

  • A deterministic "Main" protocol = an NTP clock ticking at a fixed interval (e.g. 1000ms) — not a blockchain.
  • Every chain's events are mapped onto that timeline by source block height + timestamp → every node reconstructs the same order.
  • This decouples the app's tick-rate from each chain's variable block time.
The Main/NTP protocol (sync-service docs + sync/src/sync-protocols/ntp/fetcher.ts) is the official timeline. Determinism = same inputs + same code → same state on every node, because the ordering rule is fixed.
Capabilities

What do the wallets mean for partners?

  • One walletLogin() across MetaMask/EVM, Cardano CIP-30, Midnight (Lace), Mina, Polkadot, Algorand, Avail.
  • Full sync + batcher + wallet on EVM, Midnight, Cardano, Avail, Bitcoin; others are wallet-connect / sign-only today.
  • Users keep L1 custody throughout — partners integrate without holding funds.
@effectstream/wallets WalletMode covers 8+ wallet families. Partners get one login + sendTransaction abstraction across ecosystems; the non-custodial model means they never take custody. Full-sync vs sign-only differs by chain (see the chains support matrix) — worth setting expectations per partner.
